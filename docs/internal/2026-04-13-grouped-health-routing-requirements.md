# 分组健康路由改造需求文档

- 日期：2026-04-13
- 状态：待评审
- 基线：以 `origin/main` 为准，不参考当前工作区上方的本地提交

## 1. 背景

当前系统已经支持两类基础路由能力：

- `routing rule`：根据条件选择目标 `provider/model`
- `fallbacks`：主目标失败后，按顺序尝试备用 `provider/model`

但当前系统还不具备以下能力：

- 不能在请求开始前，根据健康状态跳过异常目标
- 没有“分组”这个路由层级
- 不能在“主分组不可用”时自动切到备用分组
- 没有独立页面查看分组健康状态
- 单次请求虽然已有路由日志承载位置，但缺少分组级过程信息

本次改造目标是：在尽量少改动原始架构的前提下，把“分组 + 健康检测 + 自动切组 + 路由过程可见”整合进现有路由规则体系。

## 2. 目标

### 2.1 业务目标

- 让一条路由规则可以配置多个有序分组
- 每个分组内可以放多个候选目标
- 系统在请求开始前跳过当前处于冷却期的异常目标
- 当前分组无可用目标时，自动切到下一个分组
- 当前分组有可用目标时，只在当前分组内消耗这一组的重试预算
- 组内重试预算用完后，再切到下一组
- 在日志详情中看到每次请求的完整智能路由过程
- 提供一个页面查看当前分组和目标的健康状态

### 2.2 技术目标

- 不破坏现有 `routing rule + fallback` 基础能力
- 对旧规则保持兼容
- 新能力优先复用现有日志、规则和页面入口
- 第一阶段不引入复杂外部依赖，不做数据库持久化健康状态

## 3. 非目标

本次不做以下内容：

- 不承诺“绝对 100% 一定成功”
- 不自动修改请求内容以换取成功率
- 不做跨实例共享健康状态
- 不做 Redis 锁、分布式探活协调、数据库持久化健康状态
- 不做复杂的异常分类策略
- 不把健康状态页扩展成完整运维控制台
- 不改写现有日志主页面结构

说明：在以下情况下，系统仍然可能返回错误：

- 所有分组都失败
- 请求本身不合法
- 所有候选模型都不支持该请求参数
- 上游整体性故障或统一限流

本次目标是“尽量提高成功率并尽量少报错”，不是“逻辑上绝不失败”。

## 4. 已确认的产品决策

以下内容已按当前讨论结果固定为本期方案：

| 事项 | 决策 |
|---|---|
| 分组切换方式 | 多个分组按固定顺序切换 |
| 分组配置位置 | 放在 `routing rule` 内部 |
| 当前分组全异常时 | 直接切到下一分组 |
| 当前分组有可用目标时 | 只使用可用目标，且只在当前分组内重试 |
| 重试预算 | 每个分组共享一笔重试次数 |
| 分组用尽后 | 切到下一分组 |
| 请求内容降级 | 不允许，只能换分组 |
| 健康判定粒度 | `站点 + 模型` |
| 异常跳过策略 | 参考 LiteLLM：失败达到阈值后，冷却若干秒再恢复 |
| grouped target 配置 | `route_groups[].targets[]` 必须显式填写 `provider` 和 `model` |
| 组内失败后的动作 | 不对同一 target 做原地重试，只改选当前组内其它 target；本组无其它可用 target 或预算耗尽才切组 |
| 页面方向 | `Adaptive Routing` 只看健康状态；请求过程继续复用日志详情页 |

## 5. 推荐的完整方案

## 5.1 总体思路

将现有单层 `routing rule -> targets/fallbacks` 扩展为：

- 老规则：继续按原逻辑运行
- 新规则：可选启用“分组健康路由”

当某条规则启用“分组健康路由”后：

1. 规则命中
2. 进入第一个分组
3. 过滤掉该分组内处于冷却期的目标
4. 如果这一组还有可用目标，则在这一组内按权重选目标并消耗该组重试预算
5. 如果这一组没有可用目标，或重试预算用尽仍失败，则切到下一组
6. 任一分组成功即返回结果
7. 所有分组都失败时，返回最终错误

## 5.2 为什么选这个方案

优点：

- 保留原系统的规则命中方式，不重做整个路由链
- 分组逻辑只接管“命中规则后的目标选择”
- 健康跳过与分组切换边界清晰，行为容易解释
- 可以最大程度复用现有日志详情页
- 兼容旧规则，迁移成本低

代价：

- 路由规则的数据结构会增加一层
- 单次请求的日志会更长
- 最坏情况下，请求总耗时会增加

## 6. 详细需求

## 6.1 路由规则数据结构

为保持兼容，不直接替换旧字段，而是新增可选字段。

### 6.1.1 现有字段保留

继续保留：

- `targets`
- `fallbacks`
- `scope`
- `priority`
- `enabled`
- `cel_expression`

### 6.1.2 新增字段

建议在 `routing rule` 上新增：

```json
{
  "grouped_routing_enabled": true,
  "health_policy": {
    "failure_threshold": 2,
    "failure_window_seconds": 30,
    "cooldown_seconds": 30
  },
  "route_groups": [
    {
      "name": "primary",
      "retry_limit": 2,
      "targets": [
        {
          "provider": "openai",
          "model": "gpt-4.1",
          "key_id": "..."
        },
        {
          "provider": "openrouter",
          "model": "openai/gpt-4.1",
          "weight": 0.5
        }
      ]
    },
    {
      "name": "backup-a",
      "retry_limit": 1,
      "targets": [
        {
          "provider": "anthropic",
          "model": "claude-sonnet-4"
        }
      ]
    }
  ]
}
```

### 6.1.3 字段说明

#### `grouped_routing_enabled`

- 类型：`boolean`
- 作用：是否启用分组健康路由
- 默认：`false`

#### `health_policy`

- 类型：对象
- 建议字段：
  - `failure_threshold`
  - `failure_window_seconds`
  - `cooldown_seconds`
- 作用：定义“失败几次、在多长时间窗口内、冷却多久”

#### `route_groups`

- 类型：数组
- 作用：按顺序定义多个分组
- 数组顺序即分组切换顺序

#### `route_groups[].retry_limit`

- 类型：整数
- 语义：当前分组首轮失败后，还允许额外尝试几次
- 说明：若 `retry_limit = 2`，表示“首轮 + 2 次额外尝试”，即该组最多 3 次机会
- 补充：这些额外尝试只能改选当前组内其它可用 target，不会对刚失败的同一 target 原地重试

#### `route_groups[].targets`

- 类型：数组
- 语义：当前分组的候选目标集合
- 与普通 `targets` 不同，本期要求每个 grouped target 都必须显式填写：
  - `provider`
  - `model`
  - `key_id`
  - `weight`
- 不支持省略 `provider` 或 `model` 再去继承当前请求值

## 6.2 兼容策略

### 6.2.1 旧规则兼容

若 `grouped_routing_enabled != true` 或 `route_groups` 为空：

- 完全走现有逻辑
- `targets` 和 `fallbacks` 行为不变

### 6.2.2 新旧字段共存规则

若 `grouped_routing_enabled = true` 且 `route_groups` 非空：

- 主路由逻辑以 `route_groups` 为准
- 旧字段 `targets/fallbacks` 不参与本条规则的主选择流程
- UI 层应明确提示：该规则已切换为“分组健康路由模式”

这样可以避免旧逻辑和新逻辑同时生效导致行为混乱。

## 6.3 健康状态与冷却规则

## 6.3.1 健康判定粒度

本期按 `站点 + 模型` 记状态。

第一阶段采用以下最小标识：

- 优先使用：`provider + model + key_id`
- 若没有 `key_id`：退化为 `provider + model`

说明：

- grouped route group target 必须显式填写 `provider` 和 `model`
- 这样可以保证健康判断、健康页展示、请求日志三处使用同一份目标标识
- 若未来需要更精确的“站点实例级”隔离，可再引入显式 `target_id`

## 6.3.2 推荐默认值

文档先按推荐值落地，后续可调整：

| 参数 | 推荐值 | 原因 |
|---|---:|---|
| `failure_threshold` | 2 | 避免一次偶发失败就误伤 |
| `failure_window_seconds` | 30 | 时间窗口足够短，适合中转站波动判断 |
| `cooldown_seconds` | 30 | 能有效跳过故障目标，又不会长时间锁死 |

## 6.3.3 失败计数规则

当某个 `站点 + 模型` 失败时：

1. 记录一次失败时间戳
2. 清理窗口之外的旧失败记录
3. 如果窗口内失败次数达到阈值，则把该目标放入冷却状态

冷却状态结束后：

- 自动恢复可选
- 不需要人工干预

## 6.3.4 当前请求内的处理规则

即使某目标尚未达到全局冷却阈值，只要它在当前请求内失败了一次，也应从当前请求的候选集合中移除，避免同一次请求反复撞同一个刚失败的目标。

说明：

- 这是“单次请求内去重”
- 与“跨请求冷却”是两套机制
- 两者同时存在，行为更稳
- 对 grouped routing 命中的请求，底层 provider 自带的原地重试也应关闭；失败后直接进入当前组内其它 target 或下一组

## 6.4 组内选择与切组规则

## 6.4.1 进入分组前

进入某一分组前：

1. 读取这一组所有 targets
2. 过滤掉当前处于冷却期的 targets
3. 如果过滤后为空，则直接切到下一组

## 6.4.2 分组内尝试规则

如果当前分组还有可用 targets：

1. 按权重从可用 targets 中选择一个
2. 发起调用
3. 成功则立即返回
4. 失败则：
   - 记录该 target 当前请求失败
   - 更新该 target 的跨请求失败计数
   - 若达到阈值，写入冷却状态
   - 消耗当前分组的一次重试预算
5. 如果分组预算未用尽且还有其它可用 target，则继续组内尝试
6. 如果预算用尽，或已无可用 target，则切到下一组

补充：

- “继续组内尝试”指重新选择当前组里尚未失败、且未处于冷却状态的其它 target
- 不允许对刚失败的同一 target 做 provider 层原地重试

## 6.4.3 分组切换规则

分组按数组顺序切换，不做权重和动态优先级。

切组条件只有两类：

- 进入分组前发现该组已无可用 target
- 当前分组消耗完自己的重试预算后仍未成功

## 6.4.4 所有分组都失败时

如果所有分组都无法成功：

- 返回最终错误
- 错误中保留最后一次失败信息
- 日志中保留完整分组尝试链路

## 6.5 请求过程可见性

## 6.5.1 复用现有日志详情页

本期不新增“单次路由详情页”，而是继续复用现有日志详情页中的 `Routing Decision Logs` 区块。

原因：

- 基线已存在可用承载位置
- 前端类型和详情接口已具备基础字段
- 这是最小改动路径

## 6.5.2 必须写入的过程日志

每次请求至少需要记录以下信息：

- 命中的 `routing rule`
- 命中的分组模式是否启用
- 当前进入第几组、组名是什么
- 该组原始 target 数量
- 该组被冷却过滤掉的 target 列表
- 该组剩余可用 target 列表
- 第几次尝试选中了哪个 target
- 该次失败原因
- 该 target 当前失败计数
- 是否触发冷却、冷却到什么时间
- 当前组预算是否耗尽
- 是否切到下一组
- 最终成功使用了哪个组、哪个 target
- 若失败，最终失败停在哪一组

### 推荐日志样式

继续沿用当前纯文本日志方式，例如：

```text
[ts] [routing-rule] - Rule 'relay-chat' matched
[ts] [routing-group] - Enter group[0] primary, targets=3, available=2, cooldown=1
[ts] [routing-group] - Attempt 1/3 selected target provider=openrouter model=gpt-4.1 key=vk-1
[ts] [routing-group] - Target failed: timeout
[ts] [routing-group] - Failure count in 30s window=2, cooldown triggered for 30s
[ts] [routing-group] - Attempt 2/3 selected target provider=anthropic model=claude-sonnet-4
[ts] [routing-group] - Group primary exhausted, switching to group[1] backup-a
[ts] [routing-group] - Attempt 1/2 selected target provider=openai model=gpt-4.1-mini
[ts] [routing-group] - Request succeeded in group backup-a
```

## 6.6 健康状态页面

## 6.6.1 页面入口

复用现有侧边栏中的 `Adaptive Routing` 入口。

原因：

- 侧边栏入口已存在
- 路由与健康状态语义接近
- 比新增一个全新菜单更省改动

因此本期建议：

- 保留路径：`/workspace/adaptive-routing`
- 将当前占位页替换为真实的健康状态页

## 6.6.2 页面展示内容

页面当前展示两块：

### A. 汇总卡片

- 启用了 grouped health routing 的规则数量
- 当前可用 target 数量
- 当前处于 cooldown 的 target 数量

### B. 按规则分组的目标状态表

- 规则名称
- 该规则当前生效的健康策略参数
- 每个 target 的标识键
- 当前状态：`available / cooldown`
- 当前窗口失败次数
- 当前连续失败次数
- 冷却结束时间
- 最近一次失败原因

说明：

- 本期页面不单独展示“分组切换历史”或“最近一次切组时间”
- 单次请求的详细切组过程继续在 Logs 详情页查看

## 6.6.3 最小交互

本期只做查看，不做页面内手动干预。

不做：

- 手动清空冷却
- 手动强制拉黑
- 页面内修改规则

规则编辑仍在 `Routing Rules` 页完成。

## 6.7 配置方式

## 6.7.1 UI 配置

在现有 `Routing Rule` 编辑页中新增“分组健康路由”区域：

- 开关：是否启用分组健康路由
- 健康策略：
  - 失败阈值
  - 时间窗口（秒）
  - 冷却时间（秒）
- 分组列表：
  - 组名
  - 组内重试次数
  - 目标列表

### 交互建议

- 关闭该开关时，保留现有 `targets/fallbacks` 编辑体验
- 打开后，显示 `route_groups` 编辑器
- 旧字段区域折叠或只读提示，避免误操作

## 6.7.2 API 与存储

本期建议在 `routing rule` 结构中新增字段并持久化到现有规则存储中，不另建独立配置实体。

原因：

- 逻辑上这是规则的一部分
- 配置链路最短
- 不引入“分组模板”新概念

## 6.8 健康状态存储

## 6.8.1 第一阶段方案

采用进程内内存存储。

建议数据内容：

- 失败时间戳列表（用于窗口计数）
- 冷却截止时间
- 最近一次失败原因
- 最近一次失败时间

## 6.8.2 为什么不做数据库持久化

原因：

- 这是短周期运行态数据
- 持久化价值低
- 会显著扩大改动面
- 本期页面只需要“当前状态”，不需要长期分析

## 6.8.3 多实例限制

第一阶段明确接受以下限制：

- 不同实例之间不共享冷却状态
- 某实例判定目标进入冷却，不会自动同步到其它实例

该限制需在文档和页面说明中写明。

## 7. 与现有能力的关系

## 7.1 与现有 `fallbacks` 的关系

现有 `fallbacks` 是“主目标失败后按顺序尝试备用目标”。

新方案是“规则命中后先按分组进行目标选择和切组”。

两者差异：

| 维度 | 现有 `fallbacks` | 新分组健康路由 |
|---|---|---|
| 层级 | 单链路顺序尝试 | 分组 -> 组内目标 -> 切组 |
| 健康过滤 | 无 | 有 |
| 分组概念 | 无 | 有 |
| 重试预算 | 无组概念 | 每组独立预算，且只用于切换到同组其它 target |
| 过程展示 | 基础 fallback 信息 | 需要完整分组过程 |

## 7.2 推荐处理方式

- 旧规则继续保留 `targets/fallbacks`
- 新规则启用 `grouped_routing_enabled` 后走新路径
- 不在同一条规则中混用“组路由”和“旧 fallback 主流程”

## 8. 验收标准

## 8.1 功能验收

- 可以在一条 `routing rule` 中配置多个分组
- 每个 grouped target 都必须显式填写 `provider` 与 `model`
- 分组顺序决定切换顺序
- 当前分组全冷却时，直接跳下一组
- 当前分组还有可用目标时，不直接切组
- 当前分组共享一笔重试预算
- 同一请求内，某 target 失败后不会再对它原地重试
- 分组预算耗尽后，自动切到下一组
- `站点 + 模型` 失败达到阈值后进入冷却
- 冷却到期后自动恢复
- 日志详情能看到完整分组决策过程
- `Adaptive Routing` 页面能看到当前健康状态

## 8.2 兼容验收

- 旧 `routing rule` 不受影响
- 未启用分组健康路由的规则继续按原逻辑运行
- 现有日志、列表、筛选器不被破坏

## 8.3 可观察性验收

- 每次请求都能看到命中规则
- 每次切组都能在日志里看到原因
- 每次目标进入冷却都能看到原因和剩余时间
- 页面中显示的状态与日志中的行为一致

## 9. 风险与取舍

## 9.1 已接受的取舍

- 接受多实例之间状态不一致
- 接受极端情况下总耗时增加
- 接受第一阶段只做查看，不做手动控制
- 接受无法绝对保证“永不报错”

## 9.2 主要风险

### 风险 1：阈值过于敏感

后果：

- 正常目标因短时抖动被过早冷却

缓解：

- 默认阈值不设为 1
- 当前文档推荐 `2 次 / 30 秒 / 冷却 30 秒`

### 风险 2：目标标识不够精确

后果：

- 未配置 `key_id` 时，健康状态只能落到 `provider + model` 粒度

缓解：

- 本期先接受
- 二期如需更细，再补显式 target 标识
