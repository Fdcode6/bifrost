# 混合健康探测需求清单

日期：2026-04-14

## 1. 目标

本次改造要在现有 grouped routing 健康机制上补上“主动探测”，但不替换现有被动机制。

最终效果：

- 有真实请求结果时，优先使用真实请求产生的健康判断。
- 某个目标在一段时间内没有真实请求结果时，系统自动补一笔主动探测。
- 主动探测成功后，目标能及时恢复参与选路。
- 主动探测失败后，目标按现有 cooldown 规则进入不可选状态。

## 2. 本次必须交付

### 2.1 健康判断方式

- 保留现有被动健康机制。
- 新增主动健康探测机制。
- 两者共用同一份健康状态，不允许出现两套状态源。

### 2.2 生效范围

- 第一版只对 grouped routing 生效。
- 第一版不改全局 provider routing。
- 第一版不改系统 `/health` 接口语义。

### 2.3 主动探测触发条件

- 如果某个 grouped routing target 在配置的时间窗口内已有健康观测，不触发主动探测。
- 如果某个 grouped routing target 超过该时间窗口没有任何健康观测，触发主动探测。
- 这里的“健康观测”包含：
  - 被动成功
  - 被动失败
  - 主动成功
  - 主动失败

### 2.4 主动探测对象

- 只探测能明确定位到具体 `provider + model + key_id` 的 target。
- 以下 target 不在第一版主动探测范围内：
  - 缺少明确 `provider`
  - 缺少明确 `model`
  - 没有可用的请求类型信息

### 2.5 支持的探测类型

- 第一版支持：
  - `chat`
  - `responses`
  - `text`

- 第一版暂不支持：
  - `embedding`
  - `images`
  - `speech`
  - `transcription`
  - `batch`
  - `file`
  - 其他未明确覆盖的类型

### 2.6 被动和主动的优先级

- 被动结果优先。
- 主动探测只在目标“静默”时补位。
- 不能因为主动探测存在，就让高频真实流量目标被反复探测。

### 2.7 状态恢复

- 主动探测成功后，目标要立即恢复为可选状态。
- 不能要求目标继续等旧 cooldown 自然到期。

### 2.8 路由行为

- 继续沿用当前 cooldown 过滤逻辑。
- 继续沿用当前 fallback 逻辑。
- 当某条 grouped rule 下所有目标都不可用时，继续沿用现有兜底：
  - 跳过该 rule
  - 继续尝试后续 rule 或默认路由

### 2.9 页面展示

稳妥版把后续展示项一起纳入本次交付：

- 健康状态页增加“最后观测时间”
- 健康状态页增加“状态来源”
  - `passive`
  - `active`

要求：

- 页面能直接看出当前状态是被动还是主动更新出来的。
- 页面能直接看出最近一次健康观测发生在什么时候。

### 2.10 配置项

本次改造需要新增主动探测相关配置，放在 governance plugin 配置中。

最少包含：

- 是否启用主动探测
- 主动探测后台扫描周期
- 被动结果保鲜时间
- 单次主动探测超时
- 主动探测最大并发数

### 2.11 生命周期

- 主动探测任务只在 governance plugin 拿到可用 client 后启动。
- 服务退出时，主动探测任务必须能正常停止。

## 3. 本次不做

- 不做多实例共享健康状态
- 不把健康状态写数据库
- 不给没有明确 `provider/model` 的 target 强行猜测探测对象
- 不为不支持的请求类型临时拼凑探测方式
- 不改现有 grouped routing 的规则结构
- 不扩到所有路由场景

## 4. 成功标准

### 4.1 功能标准

- 有被动流量的目标不会被重复主动探测。
- 长时间无流量的目标会被主动探测刷新状态。
- 主动探测失败会影响 grouped routing 的可选性。
- 主动探测成功会让目标及时恢复。
- 页面能展示最后观测时间和来源。

### 4.2 安全标准

- 主动探测不会触发治理计费。
- 主动探测不会被 provider 侧重试放大。
- 主动探测不会导致选路递归。
- 主动探测不会把所有请求直接挡死。

### 4.3 可验证标准

- 有自动化测试覆盖关键健康状态变化。
- 有自动化测试覆盖 grouped routing 的核心行为不回退。
- 完成后至少验证相关测试通过。

## 5. 推荐实现边界

本次推荐按“稳妥版”实现：

- 主动探测能力
- 健康状态附加元信息
- 页面展示来源和时间
- 必要测试

不建议把多实例共享、更多请求类型支持、规则级精细配置一起塞进这一轮。
