# Logs 智能路由请求归组与最终落点分析设计稿

日期：2026-04-15

状态：待评审

关联文档：

- `docs/superpowers/specs/2026-04-15-adaptive-routing-unified-health-detection-design.md`
- `docs/superpowers/specs/2026-04-14-adaptive-routing-health-detection-ui-design.md`

说明：

- 本文档解决 `Logs` 页面中“智能路由请求切换过程难看懂、成功率被中途失败拉低、无法直观看到最终落点分布”的问题。
- 本文档重点定义“请求级”与“尝试级”两套口径，并明确首版范围，避免页面展示与统计口径相互打架。
- 本文档已经吸收本轮交叉审核意见，重点修正分页、筛选、实时更新与统计口径的冲突。

## 1. 结论

`Logs` 页面升级为“请求视图 + 尝试视图”双层观察台。

页面将新增三类信息：

- 顶部同时展示 `请求成功率` 与 `尝试成功率`
- 中部新增 `最终成功落点分布`
- 列表默认按“一次真实请求”归组显示，并支持展开查看每次尝试

本次设计采用以下固定原则：

- `Logs` 是这套能力的主入口，不把主视图放进 `Adaptive Routing`
- 同一次真实请求使用 `group_id = COALESCE(parent_request_id, id)` 归组
- `请求成功率` 只按每组最终结果计算一次
- `尝试成功率` 保留现有按单条日志统计的语义
- `最终成功落点分布` 只统计“最后真正成功承接请求”的目标，不统计中途失败尝试
- 列表首版只在最常用的“最新请求主视角”启用请求归组，不在所有排序和翻页场景强行套用
- 趋势图首版继续保留尝试级口径，不同步改成请求级
- 详情页先解释“最终结果”，再解释“切换过程”，最后才展示单次尝试细节

## 2. 背景

当前实现已经具备这些基础数据：

- fallback 请求会生成新的日志 ID
- fallback 日志会记录 `parent_request_id`
- 每条日志已记录 `fallback_index`
- 每条日志已记录 `selected_key_id`、`routing_rule_id`、`routing_engine_logs`
- 列表查询已经返回 `parent_request_id`

因此，当前缺的不是底层日志数据，而是展示层没有把同一次请求的多次尝试串起来。

同时也缺少两类前端不应该自行猜测的结构化字段：

- 哪一次尝试是这组里的第几次
- 这次尝试属于哪一层 route group

这直接带来三个问题：

1. 一次真实请求如果中间切换 2 到 3 次，页面会显示成多条相似日志，难以一眼看懂
2. 当前顶部 `Success Rate` 按尝试次数计算，会被中途失败明显拉低，不能反映真实请求最终是否成功
3. 使用者无法快速知道“流量最终都落到了哪个模型、哪个 Provider、哪一层”

因此，本次设计目标不是重做日志系统，而是在现有日志基础上补一层“请求级观察视图”：

- 让同一次请求的多次尝试能被一起看懂
- 让成功率分成“真实请求结果”与“中途尝试质量”两套口径
- 让使用者看到最终成功流量到底落到哪里

## 3. 设计目标

- 让使用者一眼看懂一次请求最后有没有成功
- 让使用者一眼看懂中间切换了几次、最终落到谁
- 让页面同时保留“真实请求结果”与“尝试质量”两套口径
- 让使用者看到最终成功流量在模型、Provider、Key、层级上的分布
- 在不重做整套日志查询架构的前提下优先交付最常用主视角
- 避免首版出现“看起来是请求视图，实际分页排序还是按尝试乱跳”的误导

## 4. 本次不做

- 不在 `Adaptive Routing` 页面做完整日志主视图
- 不把现有趋势图直接改成请求级趋势图
- 不在首版支持所有排序方式下的请求归组
- 不在首版承诺“所有翻页场景都不拆组”
- 不在首版把实时 websocket 统计改成完全请求级即时累计
- 不在首版做真正的“按请求组分页 / 按请求组搜索”后端新接口

## 5. 方案比较

| 方案 | 做法 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- | --- |
| A | 保持平铺日志，只补两个成功率与少量文案 | 改动最小 | 还是看不懂一次请求的切换过程 | 不选 |
| B | `Logs` 页面做请求归组，顶部补双成功率，再加最终成功落点分布 | 最符合使用者心智，页面价值最高 | 首版需要明确限制分页与排序范围 | 选这个 |
| C | 直接新增完整请求级后端查询、请求级分页、请求级图表 | 语义最完整 | 改动面大，首版成本高 | 作为后续演进，不做首发 |

选择方案 B 的原因：

- 直接解决“看不懂”“成功率失真”“不知道最终落点”这三个核心问题
- 现有日志字段已经够支撑首版体验
- 可以先把最常用主视角做对，再决定要不要继续做完整请求级查询体系

## 6. 页面信息架构

主入口页面：

- `/workspace/logs`

辅助入口页面：

- `/workspace/adaptive-routing`

职责分工：

| 页面 | 职责 |
| --- | --- |
| `Logs` | 查看请求级结果、尝试级质量、最终成功落点分布、请求切换过程 |
| `Adaptive Routing` | 配置路由规则与统一健康检测，不承载完整日志分析主视图 |

`Adaptive Routing` 页面只新增一个轻量入口或摘要：

- `Open in Logs`
- 最近时间范围内的请求成功率摘要
- 最近时间范围内的最终成功落点 Top 3 摘要

完整分析仍在 `Logs` 页面完成。

## 7. 页面结构设计

`Logs` 页面调整为四段式布局。

### 7.1 顶部概览区

保留顶部概览卡，但语义改成双口径。

建议卡片：

| 卡片 | 说明 |
| --- | --- |
| Request Success Rate | 一次真实请求最终是否成功 |
| Attempt Success Rate | 每次尝试单独统计的成功率 |
| Avg Final Latency | 请求最终落定那次的平均延迟 |
| Total Attempts | 当前筛选条件下的尝试总数 |
| Total Cost | 当前筛选条件下总成本 |

说明：

- 现有 `Success Rate` 改名为 `Attempt Success Rate`
- 现有 `Total Requests` 改名为 `Total Attempts`
- `Request Success Rate` 为新增请求级指标
- `Request Success Rate` 与 `Attempt Success Rate` 都必须显示“分子 / 分母 + 百分比”

示例：

- `Request Success Rate: 9,920 / 10,000 (99.20%)`
- `Attempt Success Rate: 10,880 / 15,940 (68.26%)`

### 7.2 趋势与分布区

趋势图区域调整为左右双卡。

左侧：

- 现有请求量趋势图
- 首版继续保持尝试级口径
- 文案改清楚：`Attempt Volume`

右侧：

- 新增 `Final Success Distribution`

这个新卡片默认展示：

- `By Model`

并提供以下切换：

- `By Model`
- `By Provider`
- `By Key`
- `By Layer`

每一项展示：

- 名称
- 最终成功次数
- 最终成功占比

示例：

| 维度 | 次数 | 占比 |
| --- | --- | --- |
| `gpt-4.1` | `5,200` | `52%` |
| `claude-sonnet` | `2,800` | `28%` |
| `gemini-2.5-pro` | `1,500` | `15%` |
| `fallback-open-model` | `500` | `5%` |

如果选择 `By Layer`，则直接展示：

- Layer 1 最终成功次数 / 占比
- Layer 2 最终成功次数 / 占比
- Layer 3 最终成功次数 / 占比
- Unlayered 最终成功次数 / 占比

这正对应用户最关心的：

- 第一层承接了多少
- 第二层承接了多少
- 第三层兜底了多少

### 7.3 请求归组列表

列表默认从“单条日志平铺”升级成“请求组视图”。

每个请求组代表一次真实请求。

组头展示：

| 字段 | 说明 |
| --- | --- |
| Final Status | 这次请求最终成功或失败 |
| Last Timestamp | 组内最近一次尝试时间 |
| Message Summary | 用户输入摘要 |
| Final Target | 最终落到的 Provider / Model / Key |
| Attempts | 一共尝试了几次 |
| Final Layer | 最终成功发生在哪一层 |
| Final Latency | 最终落定那次延迟 |

交互规则：

- 单次尝试的请求组看起来像普通行
- 多次尝试的请求组左侧显示展开箭头与 `N attempts`
- 点展开后，组内按顺序显示每次尝试

组内尝试子行展示：

| 字段 | 说明 |
| --- | --- |
| Attempt # | 第几次尝试 |
| Status | 成功 / 失败 |
| Provider | 本次尝试 Provider |
| Model | 本次尝试 Model |
| Key | 本次尝试 Key |
| Layer | 本次尝试所属层 |
| Latency | 本次尝试延迟 |
| Note | 失败原因或继续切换提示 |

### 7.4 详情页

详情页从“单条日志字段面板”升级成“请求摘要 + 尝试链 + 当前尝试细节”三段式结构。

#### 7.4.1 请求摘要

放在最上方，直接回答这四个问题：

- 最终有没有成功
- 最后落到了谁
- 一共尝试了几次
- 成功发生在第几层

#### 7.4.2 尝试链时间线

按顺序展示：

1. 第 1 次尝试了谁
2. 结果是什么
3. 为什么继续切换
4. 第 2 次尝试了谁
5. 结果是什么
6. 最终哪一次成功

时间线每一项显示：

- `Attempt 1`
- `Provider / Model / Key`
- `Layer`
- `Status`
- `Latency`
- `Routing decision note`

#### 7.4.3 当前尝试细节

保留现有详情页里对单次尝试的详细字段展示，例如：

- Request Details
- Tokens
- Cost
- Raw Request / Raw Response
- Routing Engine Logs

但这部分降级为“当前选中尝试的细节”，不再作为整个详情页的第一层信息。

## 8. 统计口径设计

### 8.1 请求组定义

请求组唯一标识：

- `group_id = COALESCE(parent_request_id, id)`

解释：

- 原始请求没有 `parent_request_id`，直接用 `id`
- fallback 尝试带有 `parent_request_id`，归到原始请求那一组

### 8.2 最终结果定义

一组请求的最终结果定义为：

- 该组中最后一个“已完成尝试”的结果

排序优先级：

1. `fallback_index` 更大者优先
2. 若相同，再看 `timestamp`
3. 若仍相同，再用 `id` 兜底

`processing` 状态：

- 不计入请求成功率分母
- 不作为最终结果

### 8.3 Request Success Rate

定义：

- 每个请求组只统计一次
- 如果该组最终结果为成功，则记成功
- 如果该组最终结果为失败，则记失败

字段建议：

| 字段 | 说明 |
| --- | --- |
| `completed_request_groups` | 已完成请求组数量 |
| `successful_request_groups` | 最终成功请求组数量 |
| `request_success_rate` | `successful_request_groups / completed_request_groups` |

### 8.4 Attempt Success Rate

定义：

- 保留现有成功率逻辑
- 按单条已完成日志计算

字段建议保持：

- `total_requests`
- `success_rate`

但 UI 文案统一改成：

- `total_requests` -> `Total Attempts`
- `success_rate` -> `Attempt Success Rate`

### 8.5 Final Success Distribution

定义：

- 只统计“最终成功请求组”
- 分布归属只记到最后真正成功承接请求的目标
- `By Layer` 时使用最终成功尝试所属的 route group 层级

不能采用“组内任一尝试命中就算”的原因：

- 会把中途失败目标也误算成承接成功
- 会把最终成功归给错误对象

因此，所有分布统计统一按“最终成功尝试”计算：

- `By Model`
- `By Provider`
- `By Key`
- `By Layer`

补充说明：

- `By Layer` 允许出现 `Unlayered`
- 这表示该请求最终成功了，但不是通过 grouped routing 的 route group 层级命中的

## 9. 筛选口径

请求级视图必须明确一条统一原则：

- 涉及请求成功率和最终成功分布时，只看最终尝试是否命中筛选条件

原因：

- 这是最符合使用者直觉的结果归属方式
- 不会把最终成功记到一个实际上没有承接成功的对象上

示例：

某次请求：

- Attempt 1：Model A，失败
- Attempt 2：Model B，成功

如果当前筛选是 `Model A`：

- 这次请求不计入 `Request Success Rate`
- 这次请求也不计入 `Final Success Distribution`

如果当前筛选是 `Model B`：

- 这次请求计入 `Request Success Rate`
- 这次请求计入 `Final Success Distribution`

## 10. 分页、排序与实时更新策略

这是首版最需要主动收紧范围的部分。

### 10.1 首版启用范围

请求归组视图只在以下条件下启用：

- `sort_by = timestamp`
- `order = desc`
- 默认主列表视角

其他排序方式：

- 首版保持原有平铺日志视图

原因：

- 现有后端分页与排序是按单条日志完成的
- 如果强行在所有排序场景做前端归组，会出现跨页拆组和排序值解释失真

### 10.2 组头时间

组头时间统一取：

- 组内最新一次尝试时间

原因：

- 最符合“最近活动”的直觉
- 最接近现有日志页最新在上方的行为

### 10.3 组头排序

首版请求归组视图只按：

- `latest_attempt_timestamp`

排序

不在首版支持：

- 按组 latency 排序
- 按组 tokens 排序
- 按组 cost 排序

### 10.4 实时更新

实时 websocket 到来时：

- 可以更新当前页可见请求组内容
- 不做请求级统计卡的本地增量重算
- 不做最终成功分布的本地增量重算

统一策略：

- 列表局部更新
- 统计卡与分布卡采用短延迟重新拉取

原因：

- 避免组状态、统计分母、分布占比在本地被重复累计
- 避免页面看着在变，但口径已经错掉

## 11. 详情交互与导航

为避免“展开组”和“打开详情”冲突，交互拆开：

- 左侧 `chevron` 只负责展开 / 收起
- 点击组头正文打开请求详情
- 点击子行打开该次尝试详情

上下导航规则：

- 首版按请求组切换，不按子尝试切换
- 组内尝试通过时间线点击切换

## 12. API 与数据契约调整

### 12.1 现有列表接口

现有 `/api/logs` 继续保留。

首版要求补齐并前端正式接入这些字段：

- `group_id`
- `parent_request_id`
- `attempt_sequence`
- `route_layer_index`
- `is_final_attempt`

字段说明：

| 字段 | 说明 |
| --- | --- |
| `group_id` | `COALESCE(parent_request_id, id)` 的显式返回值，前端不再自行拼接 |
| `attempt_sequence` | 这条日志是组内第几次尝试，首版按 1 开始展示 |
| `route_layer_index` | 该次尝试属于第几层 route group，首版按 1 开始展示 |
| `is_final_attempt` | 该次尝试是否是该请求组最终落定的那次 |

列表返回继续保持尝试级数据，不改成请求级分页接口。

### 12.2 统计接口

现有 `/api/logs/stats` 新增请求级字段：

| 字段 | 说明 |
| --- | --- |
| `completed_request_groups` | 已完成请求组数 |
| `successful_request_groups` | 最终成功请求组数 |
| `request_success_rate` | 请求成功率 |

说明：

- 这部分不能只靠前端当前页归组计算
- 必须由后端基于完整筛选结果计算

### 12.3 最终成功分布接口

新增独立接口：

- `/api/logs/final-distribution`

支持参数：

- 与当前 `Logs` 页相同的筛选条件
- `group_by=model|provider|key|layer`

返回结构建议：

| 字段 | 说明 |
| --- | --- |
| `dimension` | 当前统计维度 |
| `items[].value` | 稳定值，供前端识别 |
| `items[].label` | 展示名称 |
| `items[].success_count` | 最终成功次数 |
| `items[].success_ratio` | 最终成功占比 |

### 12.4 请求详情辅助接口

首版可以先不新增“按组查详情”接口。

前端做法：

- 当前页列表先按可见记录归组
- 详情页优先展示当前页已有组内容
- 若后续需要完整请求组详情，再追加按组查询接口

## 13. 性能与风险

### 13.1 已知限制

- 现有请求归组列表仍然建立在尝试级分页上
- 跨页拆组问题首版通过“只在默认主视角启用归组”规避
- 现有 materialized view 不能直接支持请求级最终结果统计

### 13.2 后端性能建议

如果请求级统计后续成为高频主视图，建议补两件事：

1. 为 `parent_request_id` 增加索引
2. 后续视情况补真正的请求级聚合或请求级分页接口

### 13.3 首版风险控制

首版通过以下策略降低风险：

- 请求归组只在默认时间排序主视角启用
- 趋势图首版继续保持尝试级
- 请求成功率与最终分布由后端统一计算
- 实时更新只更新列表，不在前端本地硬算请求级统计

## 14. 验收标准

交付后，页面应满足以下结果：

1. 使用者可以清楚区分“请求成功率”和“尝试成功率”
2. 一次真实请求发生多次切换时，列表能一眼看出最终成功、最终落点和尝试次数
3. 详情页能清楚解释每次尝试顺序以及最终为什么成功或失败
4. 使用者能直观看到最终成功流量主要落在哪些模型、Provider、Key 或层级
5. 首版不会在常用主视角里出现明显的分页错乱、排序错乱或统计口径误导

## 15. 后续演进

如果首版使用频率高，下一阶段按以下顺序推进：

1. 新增真正的请求级分页与搜索接口
2. 把趋势图同步升级成请求级与尝试级双口径切换
3. 扩展请求级详情查询，支持跨页完整请求链
